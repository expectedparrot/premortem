Structured Pre-mortem Failure Analysis

From a vivid imagined failure through stakeholder perspectives and causal pathways to concrete mitigations, research questions, and a decision-ready report.

Tool: PremortemMethod: prospective hindsightAudience: decision makers, analysts, and agents

01 Orientation

A pre-mortem assumes that an initiative has already failed and works backward to explain how. This prospective-hindsight frame gives participants permission to surface concerns that ordinary planning can suppress.

Premortem turns that exercise into an auditable workflow. It separates divergent discovery from causal synthesis and keeps recommendations attached to the failure mechanisms they are meant to change.

Imagine

Describe definitive failure as a completed fact, including visible consequences.

Diversify

Use distinct stakeholder lenses to expose different incentives, information, and pain points.

Explain

Organize failure stories into a compact causal graph of roots, effects, and outcomes.

Intervene

Target dangerous pathways with mitigations and research that can change the decision.

A decision aid, not a prediction. The objective is not to estimate one inevitable future. It is to identify plausible mechanisms early enough to prevent them or test the assumptions behind them.

JSON output contract

Every command emits one JSON document by default. Automation should branch on the boolean ok, read successful results from data, and use next_actions to discover executable follow-up commands. The command array records the real invocation.

{
  "schema_version": "1.0",
  "ok": true,
  "command": ["status"],
  "data": {
    "initiative": "Product launch",
    "phase": "personas",
    "personas": 0,
    "graph_nodes": 0,
    "mitigations": 0
  },
  "warnings": [],
  "next_actions": []
}

Failures use the same outer contract, replace data with a structured error, and return a nonzero process status:

{
  "schema_version": "1.0",
  "ok": false,
  "command": ["docs", "show", "missing"],
  "error": {
    "code": "ID_NOT_FOUND",
    "message": "Documentation topic not found.",
    "details": {
      "context": "missing",
      "hint": "Run `premortem docs list`."
    }
  },
  "warnings": [],
  "next_actions": []
}
Human output is explicit. Add --human when reading interactively. JSON remains the stable default for agents, scripts, and tests.

02 Define failure

Begin with the initiative, its context, and the outcomes that must be avoided. Then state failure in the past tense as though the team is looking back from the future.

A strong statementA weak statement
Names a date or horizon.Uses an indefinite “someday.”
Describes observable consequences.Says only that the project “did not go well.”
Leaves causes for the analysis.Embeds a favored causal explanation.
Avoids hedging.Uses “might,” “could,” or “possibly.”
Example. “It is six months after launch. Adoption remains below 10%, support costs have doubled, and three key customers have left.” The statement is vivid and measurable without claiming why it happened.
premortem init --initiative "Product launch" \
  --failure "It is six months after launch. Adoption remains below 10%, support costs have doubled, and three key customers have left."

The command returns the stored project record and a structured next action:

{
  "schema_version": "1.0",
  "ok": true,
  "command": ["init", "--initiative", "Product launch", "--failure", "It is six months after launch..."],
  "data": {
    "id": "pm_2026_0723120000",
    "initiative": "Product launch",
    "failure_statement": "It is six months after launch...",
    "phase": "personas"
  },
  "warnings": [],
  "next_actions": [{
    "label": "Run `premortem persona add`",
    "command": ["premortem", "persona", "add"],
    "mutates_state": true,
    "requires_network": false,
    "requires_user_approval": false
  }]
}
Approval checkpoint. Show the complete failure statement to the decision owner and get explicit approval before initializing the project.

03 Stakeholder personas

Create four to six perspectives whose incentives, access to information, and exposure to consequences differ. Useful roles often include the builder, buyer or user, operator, skeptic, and decision owner.

Personas are analytic lenses, not decorative biographies. Each one should reveal failure mechanisms that another role may miss.

premortem persona add --name customer_admin --role "Customer administrator"
premortem persona add --name support_lead --role "Support lead"
premortem persona add --name sales_owner --role "Enterprise sales owner"

# Or generate a set from project context:
premortem analyze personas \
  --context "B2B SaaS product-launch facilitator" \
  --requirements "customer admin,end user,support lead,sales owner,engineering lead"
premortem persona list

A list command places the collection directly in data:

{
  "schema_version": "1.0",
  "ok": true,
  "command": ["persona", "list"],
  "data": [{
    "id": "p001",
    "name": "customer_admin",
    "role": "Customer administrator",
    "perspective": null
  }],
  "warnings": [],
  "next_actions": []
}
Approval checkpoint. Review the persona set before generating reasons. Replace generic or redundant roles and check that consequential outsiders are represented.

04 Failure reasons

Ask each persona to explain how the failure unfolded. Collect both episodic event chains and structural conditions. Concrete mechanisms are more useful than abstract risk labels.

Generic labelOperational mechanism
Poor communicationSupport learns of breaking workflow changes from customers after deployment.
Lack of alignmentSales promises a migration path that engineering did not resource.
Low adoptionAdministrators cannot map existing permissions during onboarding and abandon setup.
premortem analyze reasons \
  --domain "permission model, migration tooling, support capacity, launch timeline" \
  --good-example "Administrators cannot map legacy custom roles during setup, so pilot accounts stall before inviting users."
premortem reason list
premortem status

If the output reads like a generic risk register, regenerate with stronger domain context and one example at the desired level of specificity.

05 Causal graph

Synthesize the reasons into a small directed graph: roughly three or four root causes, two or three intermediate effects, and one or two terminal outcomes. Edges should name causal mechanisms, not merely imply sequence.

Root causes

No incoming edges. These are upstream conditions that create or amplify the pathway.

Intermediate effects

Where causes combine, feedback develops, or symptoms begin to propagate.

Terminal outcomes

No outgoing edges. These connect directly to the approved failure statement.

Convergence points

Nodes reached by multiple pathways. These are often high-leverage intervention targets.

premortem graph add-node \
  --label "Permission migration blocks setup" --reason r001
premortem graph add-node \
  --label "Administrators abandon onboarding" --reason r002
premortem graph add-edge --from n001 --to n002 \
  --label "failed setup attempts"
premortem graph list

Graph output remains structured rather than embedding a rendered table:

{
  "schema_version": "1.0",
  "ok": true,
  "command": ["graph", "list"],
  "data": {
    "nodes": [
      {"id": "n001", "label": "Permission migration blocks setup", "reason_ids": ["r001"]},
      {"id": "n002", "label": "Administrators abandon onboarding", "reason_ids": ["r002"]}
    ],
    "edges": [
      {"id": "e001", "source": "n001", "target": "n002", "label": "failed setup attempts"}
    ]
  },
  "warnings": [],
  "next_actions": []
}
Approval checkpoint. The graph is analyst work. Review it for missing mechanisms, overloaded nodes, and ambiguous edges before moving to mitigation.

06 Scoring risks

Score important graph nodes to direct attention, not to manufacture false precision. Severity indicates consequence; controllability indicates leverage; uncertainty identifies where research has value; evidence records how much confidence the team should place in the claim.

DimensionQuestion
SeverityIf this node occurs, how much does it contribute to definitive failure?
ControllabilityCan the team materially change it before commitment or launch?
UncertaintyHow unsure are we that the mechanism or condition is real?
EvidenceWhat observations, records, or precedents support the assessment?
premortem score set --node n001 --likelihood high --impact high
premortem status

07 Mitigations

Every mitigation should name the graph node it targets and specify who does what by when. Separate prevention, early detection, and response; they intervene at different points in the pathway.

Concrete form. “Before the pilot begins, the customer-success lead will test permission migration with five existing administrators and publish the observed completion and abandonment rates.”
premortem analyze mitigations \
  --good-example "Before the pilot, customer success tests role migration with five administrators and records completion and abandonment rates."
premortem mitigate list

A mitigation with no graph target is probably too broad. Map it to a mechanism or treat it as a weak recommendation requiring revision.

08 Research agenda

Not every uncertainty should become an action item. Research is valuable when an assumption is important, uncertain, and testable before the decision. A useful research question identifies the evidence that would update the recommendation.

premortem analyze research-agenda
QuestionDecision relevanceEvidence
Can current administrators complete permission migration unaided?Determines whether self-serve onboarding is viable.Observed completion rate and failure points in five moderated tests.
Does the proposed workflow increase support burden?Determines staffing and rollout pace.Tickets per pilot account compared with the current product.

09 Reporting

The final report should lead with the recommendation, then show the failure definition, stakeholder lenses, causal pathways, prioritized mitigations, and highest-leverage research. Preserve provenance so readers can distinguish user input, generated material, analyst synthesis, and evidence.

premortem analyze summary
premortem report generate
premortem agent-end

Artifact-producing commands return the path and useful counts in data:

{
  "schema_version": "1.0",
  "ok": true,
  "command": ["analyze", "report"],
  "data": {
    "format": "html",
    "output_path": ".premortem/output/report.html",
    "persona_count": 5,
    "reason_count": 10,
    "graph_node_count": 8,
    "graph_edge_count": 10
  },
  "warnings": [],
  "next_actions": []
}

The handoff should name the canonical report, make ownership explicit, and state what evidence must be gathered before the next go/no-go decision.

10 Canonical command sequence

# Install locally
python -m pip install -e .

# Initialize only after the failure statement is approved
premortem init --initiative "Product launch" \
  --failure "It is six months after launch. Adoption is below 10%."

# Generate, inspect, and advance through the workflow
premortem analyze personas \
  --context "B2B SaaS product-launch facilitator" \
  --requirements "customer admin,end user,support,sales,engineering"
premortem ingest personas --from .premortem/output/results_personas.json
premortem analyze reasons \
  --domain "permissions, migration, support, sales commitments" \
  --good-example "Administrators cannot map legacy custom roles, so pilot accounts stall before inviting users."
premortem ingest reasons --from .premortem/output/results_reasons.json
premortem graph add-node --label "A concrete cause" --reason r001
premortem graph add-node --label "A downstream effect" --reason r002
premortem graph add-edge --from n001 --to n002 --label "causal mechanism"
premortem score set --node n001 --likelihood high --impact high
premortem analyze mitigations \
  --good-example "Before the pilot, customer success tests role migration with five administrators."
premortem analyze research-agenda
premortem analyze summary
premortem report generate

# Check state whenever resuming
premortem status
premortem workflow
Use the CLI as the source of truth. Project state lives in .premortem/. Do not edit its JSON files directly; use commands so validation and workflow state remain consistent.
